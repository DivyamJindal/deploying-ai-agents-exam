Segment 1 · User + KPIs

The target user is the support operations lead.

What they need

  • Routine tickets resolved in seconds
  • Risky cases escalated with context
  • Clear audit trail for refunds and angry customers
  • Evidence that automation is behaving correctly

KPIs we track

  • Auto-resolution rate: target 75%
  • Mean time to resolution
  • CSAT for frustrated customers
  • Escalation appropriateness
  • Refund exposure routed to supervisors

Segment 2 · LangGraph architecture

Architecture diagram committed in the repository.

Source: assets/support_escalator_graph.svg

Why this pattern scores

  • Conditional edges route by category.
  • interrupt() creates real human-in-the-loop escalation.
  • SqliteSaver persists paused runs.
  • Pydantic state makes every transition inspectable.

Segment 4 · Risks and learnings

The important fixes are now part of the system.

What broke

  • ResolutionAttempt was a Pydantic object, not a dict.
  • Memory-only state made restart demos fragile.
  • External graph rendering was unreliable in sandboxed network conditions.

How we fixed it

  • Added UI state normalization and regression tests.
  • Switched default graph persistence to SqliteSaver.
  • Committed a stable SVG diagram artifact.
