Segment 2 · 2 min · Architecture — pattern choice

Why LangGraph, not a single chatbot.

Single chatbot

  • Conditional routing buried in a prompt
  • No clean way to pause for a human
  • State is implicit, untyped, easily lost
  • "Resume" means replaying the full chat
  • Sentiment + escalation rules drift together

LangGraph (this build)

  • Explicit nodes — classify / solve / monitor / escalate
  • interrupt() pauses + persists waiting for the human
  • SqliteSaver checkpoints survive process restarts
  • Command(resume=…) picks up at the exact node
  • Pydantic SupportState — one schema, one source of truth

required pattern   Human-in-the-Loop + Conditional Routing — the rubric's most-asked combination.

Segment 2 · 2 min · Architecture — typed state

One Pydantic schema, end to end.

class SupportState(BaseModel):
    ticket: TicketInput
    category: Literal['bug','billing','feature','general'] | None = None
    resolution_attempts: list[ResolutionAttempt] = []
    sentiment_score: float = 0.0
    escalation_reason: str | None = None
    supervisor_input: str | None = None
    final_response: str | None = None
    ticket_metadata: dict[str, str] = {}

Every node returns a partial state dict. LangGraph merges. Pydantic validates. category is a Literal — invalid transitions can't compile.

Segment 4 · 1.5 min · Edge cases + what broke & how we fixed

The interesting failures.

Edge cases handled by design

  • Missing account — solver records the gap, gate escalates
  • Unresolved bug — solver returns resolved=False; gate trips
  • Angry tone — sentiment ≥ 0.67 always escalates
  • Refund > $200 — hard cap; LLM never auto-fires it
  • Supervisor rejects — ticket stays open, guidance attached
  • LLM down or no key — rule-based fallback keeps the graph working

What broke. How we fixed it.

  • Bug: 'ResolutionAttempt' object is not subscriptable in Streamlit. Fix: added ui_state.to_plain() normalization layer + regression tests so any Pydantic / dict / Interrupt renders safely.
  • Bug: demo state lost on every Streamlit rerun. Fix: swapped MemorySaver for SqliteSaver with a per-session thread_id the sidebar surfaces.
  • Bug: Reveal.js painted the deck white over our navy theme. Fix: CSS overrides on .reveal-viewport, .reveal .slides, etc. with !important.
  • Bug: mermaid.ink blocked from sandbox during graph PNG render. Fix: committed an authored SVG so the artifact is always present.

Segment 5 · 0.5 min · Close — limitations & what's next

What we'd ship next.

Honest limitations

  • KB is a small synthetic corpus — production needs RAG over real docs
  • Single-tenant SQLite checkpointer — Postgres at real scale
  • One supervisor seat — production needs a queue with assignment
  • No SLA enforcement yet — sidebar control wired, logic stubbed

Next steps

  • SLA timers per category with breach-triggered escalation
  • Zendesk webhook adapter (one-way ingest first)
  • Multilingual response drafting via the same LLM node
  • CSV / PDF export from the Inbox for ops audits
  • LangSmith traces for production observability

End-to-end: <200ms without an LLM, ~2s with gpt-4.1-nano. The supervisor is the long pole — and that's correct.

Likely Q&A targets

  • Why this pattern? Routing and interrupts are both required — categories fan out, escalation is human-gated.
  • What if a node fails? Partial state is preserved in resolution_attempts; production wraps each solver in retry + escalates unresolved tickets.
  • How does state prevent invalid transitions? Pydantic Literal on category, structured ResolutionAttempt, normalized via to_plain before the UI.
  • What if the supervisor rejects? Final response carries the rejection guidance and the ticket stays open.
  • How do you measure auto-resolution? Analytics tab calls summarize_run over the run history live.
